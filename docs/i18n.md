# Internationalization (i18n) & RTL

Nodexia ships a bilingual UI — **English (en)** and **Persian / Farsi (fa)** —
with a header language switcher and full right-to-left (RTL) support for
Persian. The mechanism is locale-agnostic: adding a third language is a JSON
file plus a one-line registration, no code surgery.

> **Status (phase 3).** Every `.gohtml` template is translated, **every module
> placeholder title/description resolves through i18n**, **the language switcher
> is reachable on mobile** (in the drawer), and **the client-side JS layer is
> localized**. The shell plus these pages are fully bilingual (template strings +
> handler-set titles/descriptions + flash + error pages + JS): home, **servers**
> (list + form), **monitoring**, **analytics** (per-server + global), **alerts**
> (overview + rule/channel forms), **nodes** (discovery + install), **bulk**,
> **commands**, **terminal**, **files**, **system**, **diagnostics**, **login**,
> and the shared error/placeholder pages. Each is exercised in both locales by
> `internal/view/i18n_pages_test.go`, and `en`/`fa` key parity (820 keys) is
> enforced by `TestCatalogKeyParity`.
>
> **Module page titles.** A module definition is registered once at startup
> (static), but a page renders per request (a language is known only then). So
> `module.PlaceholderPage` carries a `TitleKey`/`DescriptionKey` (catalog keys,
> not literals) and `NewPlaceholderHandler` resolves them against the request's
> localizer — the handler closure is the per-request seam. The real (DB-available)
> handlers already set their titles via `page.T(...)`.
>
> **Client-side JS strings.** The server serialises only the keys the JS needs
> (`internal/view/clienti18n.go` → `clientI18nKeys`) into a **non-executable**
> `<script type="application/json" id="nodexia-i18n">` island in the layout. That
> island is data, not script, so it is exempt from the strict `script-src 'self'`
> CSP (no inline JS is added). `app.js` parses it once and exposes
> `window.nxT(key, params)` / `window.nxTn(key, count, params)`, mirroring the
> template `{{ t }}`/`{{ tn }}` (plural by count, `{name}` substitution, key as
> fallback). Every other `*.js` file reads strings through those helpers. The
> whole catalog is **never** shipped — only the ~110 client keys. `clientI18nKeys`
> is guarded by `internal/view/clienti18n_test.go`.
>
> **Remaining (tracked follow-ups)** — none break either language; the English
> text simply isn't keyed yet:
> - **Strings built in pure Go without request access:** node action labels
>   (Restart/Update/Renew cert/… in `internal/module/nodes` providers — the
>   `nodes.cert_missing` hint references the still-English "Renew cert" button),
>   form-field *validation* messages added inside `validation.go` validators, and
>   a few `alerts/view.go` data labels (metric labels, "All enabled channels",
>   "Never"). These need a localizer threaded through their builders.
> - **Service worker (`web/static/sw.js`)** strings ("You are offline." 503 body,
>   default push title) and the static `offline.html`: the SW runs without DOM
>   access, so it cannot read the i18n island — localizing it needs a separate
>   mechanism (cache the active locale) and is deferred.
> - **Form `placeholder=""` examples** (`uname -a && uptime`, `prod, eu, rebecca`)
>   are illustrative sample input (machine-ish), intentionally left untranslated.

## Why this approach

The translation layer is a **custom, dependency-free package** (`internal/i18n`)
that loads per-language JSON catalogs embedded at build time and resolves a
dotted key (e.g. `nav.servers`) to a string in the active language.

We deliberately did **not** adopt `golang.org/x/text/message` or
`nicksnyder/go-i18n`:

| Option | Why not |
| --- | --- |
| `golang.org/x/text/message` | Powerful CLDR plurals, but the catalog/`gotext` codegen workflow is heavyweight and its API is keyed on English source text rather than stable keys. |
| `nicksnyder/go-i18n` | Nice TOML/JSON catalogs, but adds a dependency (and a TOML parser) for features we can cover in ~250 lines. |
| **Custom JSON map (chosen)** | Zero new dependencies, per-language files, pluralization, and drop-in new locales — fits the project's "few dependencies, lean SSR" constraints exactly. |

Everything is **offline-friendly**: locale JSON is embedded via `go:embed`, and
the Persian font (Vazirmatn) is self-hosted under `web/static/fonts/`. Nothing
is fetched from a CDN at runtime.

## How it fits together

```
internal/i18n/
  i18n.go        Bundle, Localizer, plural rules, T/Tn
  resolve.go     cookie + Accept-Language resolution
  context.go     stash/fetch the request Localizer in context.Context
  locales/
    en.json      default language (must exist)
    fa.json      Persian
internal/http/middleware/locale.go   resolves the request language → context
internal/http/handlers/lang.go       GET /lang/{code} switcher endpoint
internal/view/render.go              binds {{ t }}/{{ tn }}/clientI18nJSON per render
internal/view/clienti18n.go          client-needed keys → JSON island for the JS
web/templates/*.gohtml               use {{ t "key" }}
web/static/app.js                    parses the island → window.nxT / window.nxTn
web/static/style.css                 logical properties + RTL adjustments
```

### Request lifecycle

1. **`middleware.Locale`** runs per request: it reads the `nodexia_lang`
   cookie, falls back to the `Accept-Language` header, then to English, and
   stores a bound `*i18n.Localizer` in the request context.
2. **`view.NewPageData(cfg, r)`** pulls that localizer out of the context and
   attaches it to the page, deriving `Lang`, `Dir`, and the switcher options.
3. **`Renderer.Render`** clones the parsed template set and binds the `t` / `tn`
   template functions to the page's localizer, then executes. Cloning per
   render is what lets `{{ t "key" }}` resolve the right language **anywhere**,
   including inside `{{ range }}` blocks, without threading the data dot.
4. The layout emits `<html lang="{{ .Lang }}" dir="{{ .Dir }}">`.

### Locale resolution & persistence

Priority order (`Bundle.Resolve`):

1. The `nodexia_lang` cookie — an explicit, still-supported choice.
2. The best `Accept-Language` match on first visit (matched on the primary
   subtag, so `fa-IR` matches `fa`, honouring `q=` weights).
3. The default language, **English**.

The switcher (`GET /lang/{code}`) persists the choice in `nodexia_lang`
(SameSite=Lax, 1-year) and redirects back to a validated same-origin path.
Switching is a safe UI preference, so the route is GET and CSRF-exempt; the
redirect target is sanitized to block open redirects.

### Fallback

`Localizer.T` resolves **active language → default language → the key itself**.
A key missing from `fa.json` renders the English string; a key missing
everywhere renders the raw key (visible, never blank), so gaps are obvious in
review.

## Using translations in templates

```gotemplate
{{ t "nav.servers" }}                         {{/* simple string */}}
{{ t "shell.meta.env" "value" .Environment }} {{/* named placeholder {value} */}}
{{ tn "home.server_count" .Total }}           {{/* pluralized; {count} is auto-filled */}}
```

- `t key [name value ...]` — substitutes `{name}` markers from alternating
  name/value pairs.
- `tn key count [name value ...]` — picks the plural form for `count` in the
  active language; `{count}` is always available.

In Go handlers you can localize values you put on `PageData` (e.g. the page
title) with the same semantics:

```go
page := view.NewPageData(h.config, r)
page.PageTitle = page.T("home.page_title")
```

## Key naming convention

Keys are **lowercase, dot-separated, namespaced by area**:

```
<area>.<section>.<name>
```

- `area` — the page/feature: `nav`, `shell`, `drawer`, `home`, `servers`,
  `alerts`, `monitoring`, `nodes`, `terminal`, `commands`, `files`, `system`,
  `analytics`, `lang`, …
- `section` (optional) — a logical group within the area: `home.kpi.*`,
  `home.scheduler.*`, `home.empty.*`.
- `name` — a short, meaning-based identifier (`monitored_servers`), **not** the
  English text (`monitored-servers-label-blue`). The English wording can change
  without renaming the key.

Shared/cross-page strings live under stable shared areas (`nav.*`, `shell.*`,
`drawer.*`, `lang.*`) so they are translated once.

### Placeholders & plurals

- Placeholders are `{name}` markers, substituted by name. Keep the **same**
  placeholder names across every language for a key.
- Pluralized keys are JSON objects of CLDR categories. Implement the categories
  the language actually uses (`one`, `other` for en/fa; richer sets for Arabic,
  Russian, …):

```json
"home.server_count": { "one": "{count} server", "other": "{count} servers" }
```

Plural category selection per language lives in `pluralCategory` in
`internal/i18n/i18n.go`.

## RTL: how mirroring is achieved

The layout flips automatically because the stylesheets use **CSS logical
properties** instead of physical ones. With `<html dir="rtl">`, the browser
resolves the inline axis right-to-left, so the whole UI mirrors with no
per-direction rules:

| Physical (before) | Logical (now) |
| --- | --- |
| `margin-left` / `margin-right` | `margin-inline-start` / `margin-inline-end` |
| `padding-left` / `padding-right` | `padding-inline-start` / `padding-inline-end` |
| `border-left` / `border-right` | `border-inline-start` / `border-inline-end` |
| `left:` / `right:` | `inset-inline-start` / `inset-inline-end` |
| `text-align: left` / `right` | `text-align: start` / `end` |

In LTR these resolve **identically** to the old physical values, so English is
pixel-for-pixel unchanged.

A small set of cases cannot be expressed logically and have explicit
`[dir="rtl"]` overrides (see the bottom of `web/static/style.css`):

- **`transform: translateX(...)`** — the mobile drawer slides in via translate,
  which is physical; RTL gets a mirrored translate and box-shadow.

### What is deliberately kept LTR

These stay left-to-right regardless of UI language, by design:

- **The SSH terminal** (`.terminal-container` in `terminal.css`) —
  `direction: ltr`, monospace, `unicode-bidi: isolate`. Cursor movement, line
  wrapping, and the xterm canvas must not mirror.
- **Code / IP / port / identifier fields** — `<code>` inside fact lists and
  token lists is isolated LTR so addresses and shell snippets read correctly.
- **Charts & metric canvases** (analytics) — numeric/graphical surfaces.
- **Centering transforms** — `left: 50%` + `translateX(-50%)` is
  direction-agnostic and left physical.
- **The PWA manifest** keeps `lang: "en" / dir: "ltr"` install metadata; the
  served HTML `<html dir/lang>` is what drives the live UI direction.

### Fonts

- **English** keeps **Exo 2** (unchanged).
- **Persian** uses self-hosted **Vazirmatn** (5 static weights matching the
  400–800 scale). The swap is driven by a single CSS variable:
  `html[lang="fa"] { --font-ui: 'Vazirmatn', … }`. Vazirmatn-Regular is
  `<link rel="preload">`-ed only on the `fa` locale.

## Adding a new language

1. **Add the catalog.** Copy `internal/i18n/locales/en.json` to
   `internal/i18n/locales/<code>.json` and translate the values. Missing keys
   fall back to English automatically, so you can land a partial file.
2. **Register metadata (optional but recommended).** In `internal/i18n/i18n.go`
   add an entry to `languageMeta` (English name + endonym). If the language is
   RTL, add its code to `rtlLanguages`. Without an entry it still works (LTR,
   code shown as its own label).
3. **Plurals.** If the language's plural rules differ from English, add a case
   to `pluralCategory`.
4. **Font (RTL/complex scripts).** If the script needs a specific webfont,
   self-host it under `web/static/fonts/`, add `@font-face` blocks, and extend
   the `html[lang="<code>"] { --font-ui: … }` rule in `style.css`.

That's it — the switcher, resolution, and `<html dir/lang>` pick the language
up automatically from the embedded catalog.

## Adding / translating strings (phase 2 workflow)

For each template still carrying hardcoded English:

1. Replace the literal with `{{ t "area.section.name" }}` (or `tn` for
   counts). Use named `{placeholder}`s for interpolated values.
2. Add the key + English value to `en.json`.
3. Add the Persian translation to `fa.json` (and any other locale files).
4. If a page title/description is set in a Go handler, localize it with
   `page.T("...")`.
5. Run `go test ./...` — the render tests execute templates end-to-end and will
   fail on an undefined template func or a broken reference. Consider adding a
   per-page assertion mirroring `internal/view/i18n_render_test.go`.

### Conventions to keep

- Keys are stable identifiers; never key on the English text.
- One key per distinct user-facing string; reuse shared keys (`nav.*`, etc.).
- Keep placeholder names consistent across languages.
- Don't translate machine values: route groups, env keys, DB driver names,
  IPs/ports, command output — these render as data, not copy.

## Rich strings: `tsafe`

Most strings use `{{ t }}` (HTML-escaped). For catalog strings that contain
**trusted inline markup** (e.g. `<code>`, `<strong>`), use `{{ tsafe "key" }}`:
the catalog string is treated as trusted HTML while any interpolated
`{placeholder}` **values** are still HTML-escaped (so user data stays safe).
Only use it for strings whose markup ships in the catalog — never to wrap
arbitrary user input. Example:

```gotemplate
{{ tsafe "servers.form.agent_note" "var" "SSH_AUTH_SOCK" }}
```
```json
"servers.form.agent_note": { "en": "Nodexia uses <code>{var}</code> set at startup.", "fa": "…" }
```

## Regression guards

- `internal/i18n/parity_test.go` — `en`/`fa` must hold the same key set (and the
  same plural categories per key). Fails the build on drift.
- `internal/view/i18n_pages_test.go` — renders every translated page in `en` and
  `fa` and fails if a raw `area.key` leaks (i.e. a referenced key is missing).
  **When you translate a new page, add it (and its area prefix) to this test.**
  (It strips the client-i18n JSON island first — that island contains catalog
  keys as JSON property names by design.)
- `internal/view/clienti18n_test.go` — fails if any key in `clientI18nKeys` is
  missing from a catalog, and asserts the island serialises to valid JSON with no
  `</script>`. **When JS needs a new string, add the key here and to both
  catalogs.**
- `internal/module/module_test.go` — asserts the module placeholder resolves its
  `TitleKey`/`DescriptionKey` per request (renders the Persian title under `fa`).
